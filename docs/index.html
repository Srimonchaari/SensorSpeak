The Problem

Sensor data that
nobody can read

Industrial machines generate hundreds of thousands of accelerometer readings per minute. The raw numbers are meaningless without signal processing, classification, and a way to ask questions — and none of that comes cheap.

  • 💸

    Dedicated data science team

    Building a custom fault detection pipeline takes months of engineering time and labelled training data you don't have.

  • ☁️

    Cloud analytics subscription

    Your sensor data leaves your premises. Ongoing cost. Internet dependency. Vendor lock-in.

  • 🧠

    ML model training

    You need hundreds of labelled fault examples before training can even begin. And the model is a black box.

The Solution

Three barriers removed at once

SensorSpeak replaces all three expensive options with a single offline pipeline that any engineer can run in five minutes.

🎯

Rule-Based Detection

No training data. No ML. Deterministic threshold rules classify every sample into idle, walking, impact, shaking, or unknown — with a human-readable explanation for every decision.

🤖

Fully Local LLM

Ollama runs qwen2.5:0.5b on your machine. No data leaves your network. No API keys. No subscriptions. The 350MB model downloads once and works offline forever.

💬

Natural Language Chat

Ask "Were there any dangerous events?" in plain English. Get a grounded answer citing the exact time range, magnitude, and physical interpretation. Works for anyone on the team.

How It Works

From raw numbers to answers
in five steps

📥

Ingest

CSV upload or built-in synthetic data generator. 4 labelled segments. 19 seconds at 100Hz.

timestamp, x, y, z
⚙️

Normalise

Z-score each axis. Compute resultant magnitude √(x²+y²+z²). Rolling mean & std (0.5s window).

accel_magnitude
🔍

Detect

Four deterministic threshold rules classify each sample. Runs ≥ 10 samples become MotionEvents.

MotionEvent
📝

Summarise

One plain-English sentence per event: time range, severity label, peak magnitude, physical explanation.

VectorIndex
💡

Answer

LlamaIndex retrieves the top-4 relevant summaries. Ollama synthesises a grounded natural-language answer.

Ollama LLM
Features

Everything included out of the box

🖥️

Gradio Chat UI

Browser-based chat interface with CSV upload, backend selector, live event table, and signal visualisation. Works locally and in Colab with a public share link.

📄

Bosch PDF Ingestion

Upload any Bosch sensor datasheet (BMA400, BMI088, BMG250). Ask questions about specs and your recorded data in the same query. 1000-character overlapping chunks.

🔌

Pluggable LLM Backends

Switch between Ollama (offline), OpenAI GPT-3.5/4, HuggingFace Inference API, or HuggingFace Local — one environment variable, no code change.

🧪

85 Unit Tests

Full pytest suite covering all pipeline functions: data generation, normalisation, event detection, summarisation, query engine, and end-to-end integration.

🎛️

Fine-Tuning Dataset

One command generates a 200+ sample Alpaca-format JSONL dataset. Full instructions for PEFT+LoRA (Colab T4), Ollama Modelfile, and HuggingFace AutoTrain.

📊

3-Panel Visualisation

Matplotlib figure with raw axes, magnitude with coloured event bands, and rolling statistics with threshold lines. Saved as PNG, shown in Gradio UI.

Live Demo

Real questions. Real answers.

These responses come directly from the pipeline running on synthetic accelerometer data with the keyword fallback (Ollama not required).

Q Were there any abnormal or high-severity events?
A Two high-severity events were detected. A severe-severity "impact" event occurred from 13.00s to 14.98s (peak magnitude 16.45 m/s²) — a sharp high-amplitude spike consistent with a drop or collision. A "shaking" event of high severity was also recorded from 15.00s to 18.99s (peak magnitude 6.82 m/s²) — rapid high-frequency oscillation indicating intentional agitation.
Q How long did the device remain idle?
A The device was idle from 0.00s to 4.99s, a total of approximately 5 seconds. During this period the signal showed minimal vibration with a mean magnitude of 0.04 m/s², consistent with a stationary device at rest. No other idle periods were detected.
Q What is the operating voltage range of the BMA400? (with Bosch PDF loaded)
A According to the BMA400 datasheet (Page 12), the digital supply voltage VDD IO ranges from 1.2V to 3.6V, and the core supply voltage ranges from 1.71V to 3.6V. The device draws approximately 14.5 μA in normal power mode and as low as 0.85 μA in lowest power mode.

pdf_qa.py — upload Bosch datasheet 
from pdf_qa import build_combined_index, query_combined
from sensorspeak_core import run_pipeline

# Run the sensor pipeline first
result = run_pipeline()

# Build combined sensor + PDF index
index = build_combined_index(
    pdf_paths=['BMA400_datasheet.pdf'],
    events=result['events'],
    summaries=result['summaries'],
)

# Query specs + events together
print(query_combined(index,
    'What is the measurement range?'))

Step-by-step (Colab)

  1. 1

    Open the notebook

    Upload SensorSpeak.ipynb to Colab. Set runtime to T4 GPU for faster embeddings.

  2. 2

    Install Ollama

    In Section 6, uncomment install_ollama_colab() and run the cell. Takes ~5 min on first run.

  3. 3

    Run all cells

    Runtime → Run all. The pipeline runs automatically from data generation through to the query engine.

  4. 4

    Ask questions

    Scroll to Section 11. Edit MY_QUESTION and run the cell. Or launch the Gradio UI for a browser interface.

  5. 5

    Upload your CSV (optional)

    Uncomment the upload cell in Section 2. Must have timestamp, accel_x, accel_y, accel_z columns.

  6. 6

    Upload a Bosch PDF (optional)

    Run upload_and_build(events, summaries) from pdf_qa.py to add datasheet knowledge to the index.
